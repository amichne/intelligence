# Manifests

This directory separates current facts from future cleanup decisions.

- `source-roots.json` is the hand-authored list of local places to scan for
  skills, agents, hooks, instructions, and plugins.
- `discovered-primitives.json` is generated by
  `python3 garden/scripts/inventory-primitives.py`. It records provenance, resolved
  paths, digests, duplicate names, and duplicate content.
- `consolidation-report.json` is generated by
  `python3 garden/scripts/analyze-consolidation.py`. It separates canonical,
  authored-local, runtime, backup, and installed-marketplace entries into a
  review queue.
- `promotions.json` is hand-authored. It records primitives copied into this
  repository and the source paths they came from.
- `primitive-audits.json` is hand-authored. It records durable primitive
  quality, schema, provenance, runtime, and cleanup-readiness decisions.
- `source-review-decisions.json` is hand-authored. It records keep, cover,
  defer, retain, rewrite, and review decisions for every generated
  duplicate-name group before that group can guide trimming.
- `digest-review-decisions.json` is hand-authored. It records cover, retain,
  and review decisions for every generated duplicate-content group before that
  group can guide runtime replacement or cleanup.
- `source-root-decisions.json` is hand-authored. It records retain, promote, or
  review decisions for scanned source roots that need explicit ownership
  handling outside generated duplicate queues.
- `plugin-coverage.json` is generated by
  `python3 garden/scripts/analyze-plugin-coverage.py`. It records whether canonical
  primitives are composed by referential plugins, exposed through the
  marketplace, scoped as repository instructions, or still standalone.
- `primitive-decision-coverage.json` is generated by
  `python3 garden/scripts/analyze-primitive-decision-coverage.py`. It records the
  decision path for every discovered primitive entry and identifies unreviewed
  singleton entries that still need retain, promote, or root-level handling.
- `review-completeness.json` is generated by
  `python3 garden/scripts/analyze-review-completeness.py`. It records whether each
  canonical primitive has a durable audit entry in `primitive-audits.json`.
- `source-root-retirement.json` is generated by
  `python3 garden/scripts/analyze-source-root-retirement.py`. It records the
  root-by-root retention, replacement-readiness, and runtime-review state for
  every scanned source root.
- `source-turnoff-readiness.json` is generated by
  `python3 garden/scripts/analyze-source-turnoff-readiness.py`. It joins plugin
  coverage, review decisions, cleanup-ledger entries, and runtime-link plans
  into the current source replacement readiness state.
- `toolbox-readiness.json` is generated by
  `python3 garden/scripts/analyze-toolbox-readiness.py`. It maps the original toolbox
  objective to current manifest evidence and records whether completion is
  blocked by missing work, review-required runtime targets, or approvals.
- `runtime-activation-plan.json` is generated by
  `python3 garden/scripts/analyze-runtime-activation.py`. It turns proposed cleanup
  entries and runtime-link plans into explicit dry-run activation operations.
- `runtime-activation-preflight.json` is generated by
  `python3 garden/scripts/preflight-runtime-activation.py`. It checks dry-run
  activation operations against the current filesystem and records blockers,
  symlink targets, and paths that still need review.
- `runtime-activation-approvals.json` is generated by
  `python3 garden/scripts/analyze-runtime-activation-approvals.py`. It packages
  dry-run operations with preflight status, rollback notes, command previews,
  and approval boundaries before any runtime activation.
- `runtime-trim-execution.json` is written by
  `python3 garden/scripts/execute-runtime-trim.py` when an explicit packet-scoped dry
  run or apply attempt is made. It is a receipt, not a source of future
  authority.
- `runtime-links.json` is hand-authored. It records planned runtime symlink and
  marketplace activation targets without executing them.
- `cleanup-ledger.json` records proposed, approved, or executed cleanup. Normal
  source cleanup requires a promoted replacement and rollback path; broken
  symlink cleanup records the observed link, missing target, verification, and
  rollback note. `PROPOSED` entries do not authorize deletion or symlink writes.

Schema ownership is explicit:

- `marketplace.json`, `plugins/*/plugin.json`, and provider-neutral
  `hooks/*.hook.json` validate against `schemas/core/`.
- `hooks/codex/*.json` validates against
  `schemas/adapters/codex/hooks.schema.json`.
- `garden/manifests/*.json` validates against `garden/schemas/intelligence/*.schema.json`.
- `*.schema.json` files must compile as JSON Schema documents.

Run `python3 garden/scripts/check-source-graph.py` after changing source graph files.
Use `python3 garden/scripts/check-source-graph.py --refresh` when generated reports
should be rebuilt before validation.

Run `node scripts/validate-manifests.mjs` after changing any JSON file. The
validator rejects tracked JSON files that are not covered by a schema path.
It also requires every promotion entry to have a corresponding primitive audit
entry, so newly promoted primitives cannot skip the quality/readiness ledger.
It requires every generated name-review group to have a synchronized source
review decision, so duplicate names cannot silently become cleanup authority.
It also requires every generated digest-review group to have a synchronized
digest review decision, so byte-identical copies cannot silently become cleanup
authority.
It requires every source-root decision to point at a known scan root and current
inventory evidence, so manual root ownership decisions cannot drift silently.
It requires every discovered primitive entry to appear in the primitive decision
coverage report, so per-entry review gaps remain visible while root-level
decisions are still being made.
It requires every scanned source root to appear in the source-root retirement
report, so scan-boundary cleanup and retention state remains explicit.
For `REPLACE_WITH_SYMLINK` and `DELETE_ORIGINAL`, cleanup entries must point
back to a `COVERED_BY_CANONICAL` digest-review decision and to an existing
canonical path.
The runtime activation plan is also non-authoritative for execution: operation
previews are dry-run evidence until the relevant cleanup-ledger or runtime-link
entry is explicitly approved.
The runtime activation preflight is observational only. It may report an entry
as `READY_FOR_APPROVAL`, but that still does not authorize writes, deletion, or
symlink replacement.
The runtime activation approval queue is also non-executable. It makes approval
boundaries reviewable, but only an explicit user instruction can move a packet
from review evidence to execution.
Runtime trim execution receipts record what the packet executor attempted. They
do not approve additional packets and should not be replayed as a cleanup plan.

Do not remove source files based only on a duplicate name. Require a
synchronized source or digest review decision, a reviewed promotion entry, and a
cleanup ledger entry with a rollback path first.

Broken symlinks are a separate cleanup class: remove them only after the ledger
records `REMOVE_BROKEN_SYMLINK` and validation confirms the entry shape.

For OpenAI, Anthropic, or other first-party sources, use promotion entries as
provenance only. Canonical local primitives should be renamed and rewritten so
they do not collide with installed first-party tools. The manifest validator
also rejects canonical primitives whose content digest is identical to a
first-party source.

When a promoted primitive synthesizes several source skills, keep the canonical
name local and capability-oriented. Preserve source paths in `promotions.json`,
but keep plugin manifests referential: they should point at `skills/*`,
`agents/*`, `hooks/*`, or `concepts/*`, not copied payload folders.
